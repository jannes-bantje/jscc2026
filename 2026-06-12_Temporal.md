## `Date` was a mistake
* the `Date` API is a mess, which is the resulting of copying over `java.util.Date` in the 90s
* Java has long moved past this API, but for backwards compatibility reasons, JavaScript has not
* In the session we went through the quiz at [jsdate.wtf](https://jsdate.wtf/) together.
    * We learned that many of the quirks are in the constructor
* `Date` objects are mutable: can cause subtle errors due to passed references
* The shortcomings of `Date` has lead to a zoo of date libraries, such as `moment`, `date-fns`, `luxon` and `dayJS`. They all have their own API and are not interoperable with each other. And of course they add to your bundle size.

The Temporal API is designed to replace `Date` and is in the works since 2017.

## Conceptual differences
* Instead of a single `Date` class there is a class for every kind of date/time representation: 
  * `Instant` for a specific point in time
  * `ZonedDateTime` for a date and time in a specific timezone
  *  “plain types” such as `PlainDate`, `PlainTime`, `PlainDateTime`, `PlainMonthDay`, `PlainYearMonth` for date and time without timezone information
  * `Duration` for time differences
* All classes are immutable
* Temporal is “fail-fast” and not lenient as `Date`
* Proper timezone support
* nanoseconds precision (instead of milliseconds)

## A ~~not too~~ deep dive into the API

We played around with the API in the browser console. 
* `Temporal.Now` gives you the current date and time in different formats, such as `Temporal.Now.instant()`, `Temporal.Now.zonedDateTime()` and `Temporal.Now.plainDate()`.
* In general one should use the factory methods such as `Temporal.ZonedDateTime.from()` for creating objects.
* the `.toString()` methods of Temporal objects give you a nice ISO string representation, whereas `.toLocaleString()` can be used to 
* `Temporal.ZonedDateTime.from({year: 2026, month: 1, day: 31, timeZone: Temporal.Now.timeZoneId()}).add({months: 1})` results in 2026-02-28 (so remaining days of the month get truncated). 
* `.add({days: 1})` is not the same as `.add({hours: 24})` because of daylight saving time changes leading to some days that do not have 24 hours.

## Performance comparison

See the slides for a table comparing the performance of Temporal with the date libraries and the Temporal polyfill. Bottom line: Of course Temporal wins in both categories (bundle size and actual time performance). The temporal polyfill in the ranges in the middle field of the date libraries!

## Migration to Temporal
* Marco has done that for an application that makes heavy use of dates and times, and it was a smooth process that was done within a week (without working 100% of the time on it)
* The migration uncovered bugs in the old code!
* AI has a good understanding of the API, because the proposal exists for a long enough time.
* The decision of whether to use `PlainDate`, `ZonedDateTime` etc. for a given `Date` in the old code should _not_ be left to an AI entirely.
* The needed polyfill for Safari users is not a problem.
